# Data Engineering Pipeline

A local, fully-containerized streaming and batch data pipeline that simulates real-time e-commerce clickstream data, ingests it through Kafka, and processes it with Spark for user, product, and location-level analytics.

Built as an end-to-end sandbox for practicing the core data engineering stack: **event streaming, distributed processing, object storage, and orchestration** — all runnable on a single machine with Docker.

## Architecture

```
 ┌────────────┐      ┌───────┐      ┌────────────────┐      ┌──────────────┐
 │  Producer   │ ───▶ │ Kafka │ ───▶ │  Spark (batch/  │ ───▶ │  Console /    │
 │ (Python)    │      │ topic:│      │  streaming jobs)│      │  Postgres /   │
 │ synthetic   │      │raw-   │      │  aggregations   │      │  MinIO output │
 │ events      │      │events │      │                 │      │               │
 └────────────┘      └───────┘      └────────────────┘      └──────────────┘
                         ▲
                         │
                 ┌───────────────┐
                 │ Schema Registry│
                 └───────────────┘

 Flink JobManager/TaskManager, Jupyter (PySpark), and MinIO run alongside for
 experimentation, notebook-based development, and S3-compatible object storage.
```

Events (page views, cart adds, purchases) are generated by a Python producer and published to a Kafka topic. Spark jobs read from Kafka — in either batch or structured-streaming mode — parse and enrich the events (adding `revenue` and `event_date`), then compute aggregations: revenue and activity per user, per product category, and per location.

## Tech Stack

| Component | Role |
|---|---|
| **Apache Kafka** (+ Zookeeper, Schema Registry) | Event streaming backbone |
| **Apache Spark** (PySpark) | Batch and structured-streaming processing/aggregation |
| **Apache Flink** | Available for stream-processing experimentation (job manager + task manager) |
| **PostgreSQL** | Metadata / relational store |
| **MinIO** | S3-compatible object storage for processed data lake output |
| **Jupyter (PySpark notebook)** | Interactive exploration and development |
| **Docker Compose** | Orchestrates all of the above as a single local environment |

## Project Structure

```
data-engineering-pipeline/
├── docker-compose.yml          # All services: Kafka, Spark, Flink, Postgres, MinIO, Jupyter
├── kafka_producer.py            # Event generator — run from your host machine
├── kafka_producer_docker.py     # Same producer, configured for in-container use
├── spark-jobs/
│   ├── spark_batch_process.py   # One-shot batch read + aggregation + console output
│   └── process_kafka_data.py    # Structured-streaming version (continuous aggregation)
├── notebooks/                    # Mounted into the Jupyter container for interactive work
├── init-db/                      # SQL init scripts for Postgres (mounted on first boot)
└── data/
    ├── raw/                      # Landing zone for raw ingested data
    ├── processed/                 # Cleaned/enriched data
    ├── modeled/                   # Dimensionally modeled output
    ├── star_schema/               # Star-schema tables (fact/dimension)
    ├── checkpoints/               # Spark structured-streaming checkpoints
    └── output/                    # Final query/report output
```

## Prerequisites

- Docker and Docker Compose
- Python 3.8+ with the `kafka-python` package: `pip install kafka-python`

## Quick Start

```bash
# 1. Start all services (Kafka, Spark, Flink, Postgres, MinIO, Jupyter)
docker-compose up -d

# 2. Wait ~30 seconds for Kafka to finish initializing, then create topics
docker exec kafka kafka-topics --create --bootstrap-server localhost:9092 \
  --topic raw-events --partitions 3 --replication-factor 1
docker exec kafka kafka-topics --create --bootstrap-server localhost:9092 \
  --topic processed-events --partitions 3 --replication-factor 1

# 3. Generate synthetic e-commerce events
python kafka_producer.py
# You'll be prompted for how many events to send (default: 100)

# 4. Process the data with Spark — pick one:

## Batch: one-shot read, aggregate, and print results
docker exec spark-master /opt/spark/bin/spark-submit \
  --packages org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.0 \
  /opt/spark-jobs/spark_batch_process.py

## Streaming: continuously aggregate as new events arrive
docker exec spark-master /opt/spark/bin/spark-submit \
  --packages org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.0 \
  /opt/spark-jobs/process_kafka_data.py
```

The batch job prints sample records plus three aggregations: total events/revenue per user, per product category, and per location. The streaming job re-computes per-user event counts and revenue every 30 seconds as new Kafka messages arrive, and can be monitored live at the Spark UI below.

## Stopping

```bash
# Stop containers, keep data volumes
docker-compose down

# Stop and remove all data (Kafka topics, Postgres data, MinIO buckets, etc.)
docker-compose down -v
```
